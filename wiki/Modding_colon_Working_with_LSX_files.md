#### 文件夹结构

这里已经有教程介绍如何设置文件夹结构，总的来说，将你的模组打包成 .pak 格式而不是存档格式（用户需要将其解压到数据文件夹中）是非常有益的。BG3 使用覆盖系统，我稍后会详细介绍，将所有内容解压到数据文件夹中似乎在某种程度上规避了这种行为，因为似乎必须将内容放在 Gustav、GustavDev、Shared、SharedDev 的某种变体中。请务必仔细查看文件夹结构，因为缺少一个小细节就会导致你的模组无法加载，而且调试过程总体上很繁琐。

#### 使用 .lsx 文件，或者更准确地说 .xml 文件

根据经验水平和目标，可以采用不同的策略来修改文件。对于我的模组（进行大规模更改），我使用 python。它还帮助我查询数据并直接找到我需要的内容，而不是试图在多个位置浏览同一个文件。以编程方式操作的优势在于，如果补丁更新了你更改的相同节点，你可以重新运行脚本。在底部，我包含了用于生成我的模组的脚本，python 有点超出本教程的范围，但它应该为你提供一个骨架和处理文件的一般工作流程。所讨论的模组为所有基础职业添加了等级13，这使得你可以在一个增加最高等级的模组中达到等级12，而不会在下次升级时导致游戏崩溃。它在每次升级时提供专长，并将法术位/术法点/气点等在升级时增加三倍。

#### BG3 如何加载资源？

如前所述，了解资源如何加载和覆盖非常重要。想象一下，你解压了游戏的 .paks，浏览文件夹结构，在 Data/Public/Shared/Progressions/Progressions.lsx 中找到了等级的 xml 数据。你随心所欲地修改它，也许你删除了一些东西，添加了一些东西，然后就让它留在那个文件夹层次结构中。会发生的是，它将完全替换 .pak 文件中的那个文件。也许你注意到这个文件中只有1-6级，你认为添加7-12级是明智的。但问题是，7-12级在 SharedDev 子文件夹结构中有另一个 Progressions.lsx，它在 Shared 中的那个之后加载，并覆盖了你的7-12级。虽然这看起来令人沮丧，但它也应该让你了解系统是如何工作的。有一个加载顺序，即使在游戏附带的 .paks 内部也是如此，你通常不想打乱它，因为补丁也遵循这个加载顺序。通常，当你在路径中看到 Gustav/GustavDev/Shared/SharedDev 时，当你将其复制到你的模组项目时，应将其替换为你的模组名称。通常，你应该只包含你实际更改的文件部分，排除其他所有内容，这样游戏会填充其余部分。这为你提供了最好的机会，让你的模组在补丁到来时，或者如果另一个模组覆盖了同一文件中的内容时，仍然能够正常工作。

那么这些文件是如何合并的呢？这似乎因文件而异，例如 .txt 文件（如 Data.txt）基于键/值对的键进行合并。然而，我将重点介绍的 .lsx 文件是基于它们的 UUID 进行合并的，因此当你查看 xml 时，任何具有 UUID 子节点的组都是你将覆盖的粒度。

#### Progressions.lsx

最后，我们将介绍这个文件的结构，它对游戏中的所有升级都至关重要。重要的是要知道这个文件分布在多个位置，对于没有补丁的基础游戏，它分布在 Shared/Public/Shared/Progressions/Progressions.lsx（包含1-6级，我推测是游戏的早期访问状态）和 Shared/Public/SharedDev/Progressions/Progressions.lsx（包含7-12级和进度修改）。更改进度的补丁有自己的版本，这些版本具有优先权。作为一个经验法则，我永远不会更改任何节点的 type 属性，尽管我还没有真正尝试过。让我们分解一个职业的1级，这里我以牧师为例，因为它有很多变体，即使它非常庞大：

第一个属性，id 为 Boosts，是最重要的属性之一。它是一系列函数调用，声明了当你达到这个特定等级时实际获得的内容。在这种情况下，获得1级牧师会给我们一系列熟练项，这里可以指定的内容目前也超出了本教程的范围，但它实际上相当直接且易于推断。如果你想让你修改的升级获得其他东西，请查看文件中另一个拥有你想要的东西的升级等级，并将其附加到你的升级中。

第二个属性，id 为 IsMulticlass，是可选的，默认为 false，但在本例中为 true。这意味着这个升级实际上是用于多职业兼职牧师时的，这不是1级角色以牧师开始的规范。

第三个属性，Level，简单地指定这个声明涉及等级1。第四个属性，Name，只是职业的标识符。第五个属性我还没有推断出来，但肯定与授予的被动技能有关。

第六个属性，ProgressionType，是一个整数，决定进度的类型。0 表示它是基础职业，1 表示它是副职，3 表示它与种族相关。

第七个属性，Selectors，决定了你可以从这个升级中选择/获得什么。例如，SelectSpells(2f43a103-5bf1-4534-b14f-663decc0c525,3,0,,,,AlwaysPrepared); 以一个指向 SpellLists.lsx 的 UUID 开头，这是牧师1级戏法的集合。3 表示你可以选择3个，其他值还没有推断出来，但 AlwaysPrepared 可能如其名所示。AddSpells(269d1a3b-eed8-4131-8901-a562238f5289) 也是一个指向 SpellLists.lsx 的 UUID，这是牧师的1级法术。

第八个属性，TableUUID，是一个 UUID，它本质上将进度的所有内容绑定在一起。这也超出了本教程的范围，我还没有完全推断出来，但起点在 ClassDescriptions.lsx 内部，它将表与职业的一般描述关联起来。

第九个，UUID，只是这个升级的唯一标识符，如果你想覆盖某些东西，你必须有一个与之相同的 UUID，如果你想添加某些东西，那么它必须有一个新生成的 UUID。

最后，我们有一个子结构，它只是一个副职数组，如果你想添加一个新的副职，请复制结构并更改 UUID 值，该 UUID 映射到 ClassDescriptions.lsx 中的一个条目。

除了你期望的基础职业外，还有 MulticlassSpellSlots，这似乎只是一个占位符，当你与施法者多职业兼职时，它会启动，以确保他们根据一般角色等级获得额外的法术位。

你可能还会遇到一个 id 为 AllowImprovement 的属性，将其添加到基础职业等级并设置为 true，可以确保你在该升级时获得一个专长。
